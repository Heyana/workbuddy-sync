## 背景

- 项目：Flutter 资产管理 App (asset_manager)
- 方案：**文件级增量同步**（替代了早期的 ZIP 全包同步）
- 多账号支持：不同 WebDAV 服务器独立账号配置
- 协议：HTTP/HTTPS，基于标准 WebDAV 方法（PROPFIND / GET / PUT / DELETE / MKCOL）

## 架构决策

### 为什么从 ZIP 全量同步改为文件级增量
- ZIP 全量：每次同步打包整个数据库，大文件慢、浪费带宽、无法检测部分变更
- 文件级增量：按变更文件粒度上传/下载，支持断点续传，冲突粒度更细
- ⚠️ **教训**：目录迁移时遭遇过一次数据丢失，此后强化了同步前校验

### 同步策略
- **增量同步**：仅同步变更文件（基于时间戳 / checksum / 版本号对比）
- **方向**：双向同步（本地 ↔ 远程）

## WebDAV 协议要点

### 常用操作
| 操作 | 方法 | 用途 |
|------|------|------|
| 列出目录 | PROPFIND (Depth: 1) | 获取文件列表和属性 |
| 下载文件 | GET | 下载指定文件 |
| 上传文件 | PUT | 创建或覆盖文件 |
| 删除文件 | DELETE | 删除远程文件 |
| 创建目录 | MKCOL | 创建远程文件夹 |
| 检查存在 | HEAD | 不下载内容，仅检查存在 |

### 注意事项
- PROPFIND 响应是 XML，需解析 `DAV:` 命名空间
- 某些 WebDAV 服务器限制文件大小或并发连接数
- 大文件上传建议分块（但 WebDAV 原生不支持分块上传，需服务端配合或改为分文件上传）

## 已知踩坑

### 数据丢失
- **场景**：目录迁移期间，同步未暂停导致数据被覆写
- **根因**：迁移操作与同步流程缺乏互斥锁
- **教训**：任何文件系统级操作（迁移/重组织目录）前必须先停同步

### 数据库路径可视化
- 用户已提出需求：在 App 内显示当前数据库文件的绝对路径
- 目的：增强数据位置确认，防止"不知道数据库在哪"的困惑
